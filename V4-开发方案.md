### 阶段 1：HashedWheelTimer ACK 优化

**目标**：用 Netty HashedWheelTimer 替换 Caffeine TTL，将超时检测从 O(n) 降到 O(1)。

**数据结构变更**：
```
替换前: Caffeine Cache<String, MessagePacket> (TTL 30s)
替换后: ConcurrentHashMap<String, MessagePacket> + HashedWheelTimer(100ms, 512)
```

**核心实现**：
```java
public class PendingAckManager {
    private final HashedWheelTimer timer;          // 时间轮，单线程驱动
    private final Map<String, Timeout> timeoutMap; // key → Timeout 句柄
    private final Map<String, MessagePacket> packetMap; // key → 原始消息

    public PendingAckManager() {
        this.timer = new HashedWheelTimer(100, TimeUnit.MILLISECONDS, 512);
        this.timeoutMap = new ConcurrentHashMap<>();
        this.packetMap = new ConcurrentHashMap<>();
    }

    public void pending(String channelId, int sequenceId, MessagePacket packet) {
        String key = buildKey(channelId, sequenceId);
        packetMap.put(key, packet);
        Timeout timeout = timer.newTimeout(t -> {
            packetMap.remove(key);
            timeoutMap.remove(key);
            log.warn("消息确认超时: key={}", key);
        }, ackTimeoutMs, TimeUnit.MILLISECONDS);
        timeoutMap.put(key, timeout);
    }

    public boolean ack(String channelId, int sequenceId) {
        String key = buildKey(channelId, sequenceId);
        Timeout timeout = timeoutMap.remove(key);
        if (timeout != null) {
            timeout.cancel();     // 取消超时任务
            packetMap.remove(key);
            return true;
        }
        return false;
    }
}
```

**新增配置**：`netty.server.ack-timeout-ms: 30000`（替代硬编码）

**Checkpoint**：
- 单元测试覆盖 pending/ack/超时三条路径
- 10 万并发 pending 场景 benchmark 对比 Caffeine 内存和延迟

### 阶段 2：TCP 背压流控

**目标**：当下游 Kafka/业务队列积压时，通过 `channel.isWritable()` 暂停读取 TCP 缓冲区，利用 TCP 滑动窗口反向压制设备。

**新增文件**：`com.xsh.netty.handler.BackpressureHandler.java`

**Pipeline 位置**：`SslHandler → IpFilterHandler → ReadTimeoutHandler → BackpressureHandler → IdleStateHandler → ...`

**核心逻辑**：
```java
public class BackpressureHandler extends ChannelInboundHandlerAdapter {

    @Override
    public void channelWritabilityChanged(ChannelHandlerContext ctx) {
        // 写队列从不可写恢复为可写时，重新开启读取
        if (ctx.channel().isWritable()) {
            ctx.channel().config().setAutoRead(true);
        }
    }

    @Override
    public void channelRead(ChannelHandlerContext ctx, Object msg) {
        if (!ctx.channel().isWritable()) {
            // 暂停读取，TCP 缓冲区会满 → 滑动窗口缩小 → 设备降频
            ctx.channel().config().setAutoRead(false);
        }
        ctx.fireChannelRead(msg);
    }
}
```

**新增配置**：
```yaml
netty.server.backpressure:
  enabled: true
  high-water-mark: 65536
  low-water-mark: 32768
```

**Checkpoint**：集成测试 — 停掉 Kafka Consumer，观察网关是否停止读取设备数据。

### 阶段 3：IP 动态过滤

**目标**：在 Pipeline 前端（三次握手后立即执行）拒绝恶意扫描器的连接，降低文件描述符消耗。

**新增文件**：
- `com.xsh.netty.handler.IpFilterHandler.java`
- `com.xsh.netty.server.IpFirewallService.java`

**Pipeline 位置**：`SslHandler → IpFilterHandler → ReadTimeoutHandler → ...`

**核心逻辑**：
- `IpFirewallService`：Redis Key `ip:blacklist:{ip}` + 本地 Caffeine 缓存
- `IpFilterHandler.channelActive`：取 remoteAddress → 查本地缓存 → 强查 Redis → 命中黑名单 → `ctx.close()`
- 触发规则：60 秒内 5 次非法魔数断开 或 3 次鉴权超时断开 → 自动加入黑名单

**新增配置**：
```yaml
netty.server.ip-filter:
  enabled: true
  ban-threshold: 5
  ban-window-seconds: 60
  ban-duration-minutes: 30
```

**Checkpoint**：集成测试 — 模拟恶意扫描 6 次，验证第 6 次被拒绝。

### 阶段 4：优雅停机与离线遗言

**目标**：停机时主动通知所有设备并投递离线遗言到 Kafka，防止上层应用数据断层。

**修改文件**：`NettyServerBootstrap.java`、`DeviceChannelManager.java`

**新增 Kafka Topic**：`device-offline-events`

**核心实现**：
```java
@PreDestroy
public void stop() {
    log.info("Netty 优雅停机中...");
    // 0. 停止接收新连接
    silentClose(serverChannel, "明文 Channel");
    silentClose(tlsServerChannel, "TLS Channel");

    // 1. 向所有在线设备广播维护通知
    channelManager.broadcastMaintenanceNotice();

    // 2. 遍历所有设备 → 构造离线遗言 → 发送到 Kafka
    List<CompletableFuture<?>> futures = channelManager.emitOfflineEvents(kafkaProducerService);

    // 3. 等待遗言投递完成（超时 5s）
    CompletableFuture.allOf(futures.toArray(new CompletableFuture[0]))
            .orTimeout(5, TimeUnit.SECONDS)
            .exceptionally(e -> { log.warn("遗言投递超时"); return null; })
            .join();

    // 4. 关闭所有设备连接
    channelManager.closeAll();

    // 5. 释放 EventLoopGroup
    silentShutdown(bossGroup, "bossGroup");
    silentShutdown(workerGroup, "workerGroup");
    silentShutdown(businessGroup, "businessGroup");
}
```

**离线遗言消息格式**：
```json
{
  "deviceId": "dev-001",
  "event": "server_maintenance",
  "timestamp": 1718500000000,
  "gatewayNode": "node-1",
  "reason": "scheduled_restart"
}
```

**Checkpoint**：集成测试 — 启动网关→连接设备→关闭网关→验证 Kafka 有遗言消息→设备收到维护通知。

### 阶段 5：分布式集群连接管理

**目标**：支持多网关节点部署，设备连接路由透明，跨节点指令可达。

- `com.xsh.netty.server.ClusterRouterService.java` — Redis Pub/Sub 指令路由

**修改文件**：`DeviceChannelManager.java`

**架构**：

```
Node-1                 Node-2                 Node-3
  │                     │                     │
  ├─ DeviceChannelManager                    │
  │   ├─ localSessions (ConcurrentHashMap)   │
  │   └─ clusterRouter                       │
  │         │                                │
  └─────────┼────────────────────────────────┘
            │
      ┌─────▼─────┐
      │   Redis    │
      │ ┌────────┐ │
      │ │Hash    │ │  device:session:{deviceId} → {nodeId, connectedAt}
      │ ├────────┤ │
      │ │PubSub  │ │  cluster:command:{targetNodeId}
      │ └────────┘ │
      └─────────────┘
```

**会话注册/注销增强**：
```java
@Override
public void register(String deviceId, Channel channel) {
    // 原有逻辑：本地注册 + 踢旧保新
    DeviceSession newSession = new DeviceSession(deviceId, channel, Instant.now());
    DeviceSession oldSession = sessionMap.put(deviceId, newSession);
    if (oldSession != null) { oldSession.getChannel().close(); }
    
    // 新增：集群模式 → 注册 Redis 会话路由
    if (clusterEnabled) {
        clusterSessionManager.registerSession(deviceId, myNodeId);
    }
}

@Override
public void unregister(String deviceId, Channel channel) {
    // 原有逻辑
    sessionMap.remove(deviceId, new DeviceSession(deviceId, channel, null));
    
    // 新增：集群模式 → 清理 Redis 会话路由
    if (clusterEnabled) {
        clusterSessionManager.unregisterSession(deviceId);
    }
}
```

**sendToDevice 增强**：
```java
@Override
public boolean sendToDevice(String deviceId, Object message) {
    Channel localChannel = getChannel(deviceId);
    if (localChannel != null) {
        localChannel.writeAndFlush(message);
        return true;
    }
    // 本地无连接 → 查 Redis 会话表 → 路由到远程节点
    if (clusterEnabled) {
        String nodeId = clusterSessionManager.getNodeId(deviceId);
        if (nodeId != null && !nodeId.equals(myNodeId)) {
            clusterRouterService.routeCommand(nodeId, deviceId, message);
            return true;
        }
    }
    return false;
}
```

**Redis Pub/Sub 监听**：
```java
// Node-2 启动时订阅自己节点的指令通道
redisTemplate.getConnectionFactory().getConnection()
    .subscribe((message, pattern) -> {
        // 收到跨节点指令 → 解析 deviceId + message
        // → 查本地 Channel → writeAndFlush
    }, ("cluster:command:" + myNodeId).getBytes());
```

**新增配置**：

### 阶段 6：Modbus-TCP 协议支持

**目标**：在 `MultiProtocolDetector` 中识别 Modbus-TCP MBAP 头，实现标准 Modbus 编解码。

**新增文件**：
- `com.xsh.netty.codec.ModbusDecoder.java` — MBAP 头解析 + PDU 提取
- `com.xsh.netty.codec.ModbusEncoder.java` — MBAP 头封装 + PDU 编码
- `com.xsh.netty.handler.ModbusBusinessHandler.java` — 功能码处理

**协议探测**：
```
MBAP 头部固定 7 字节:
┌───────────────┬──────────────┬──────────┬─────────┐
│ TransactionId │  ProtocolId  │ Length   │ UnitId  │
│    (2B)       │ (2B = 0x0000)│  (2B)    │  (1B)   │
└───────────────┴──────────────┴──────────┴─────────┘
```

**嗅探逻辑**：在前 7 字节中，检查字节 2-3 是否为 `0x00 0x00`（ProtocolId = 0 = Modbus），且字节 4-5 组成的大端长度与包体一致。

**Pipeline（Modbus 路径）**：
```
MultiProtocolDetector → ModbusDecoder → ModbusEncoder → ModbusBusinessHandler
```

**Maven 依赖**：无需新增（自研编解码器，约 300 行）

**Checkpoint**：与 modpoll 标准客户端互操作，覆盖功能码 01/03/04/06/16。

### 阶段 8：动态物模型（插件接口）

**目标**：提供 `ThingModelMessageHandler` 插件接口，业务方实现异构数据→标准物模型 JSON 转换，由 DispatcherConfig 自动装配。

**新增文件**：
- `com.xsh.netty.dispatcher.ThingModelMessageHandler.java` — 接口
- `com.xsh.netty.dispatcher.ThingModelContext.java` — 上下文

**接口定义**：
```java
/**
 * 物模型转换处理器，将异构设备原始数据转换为标准物模型 JSON。
 *
 * <p>实现类通过 Spring Bean 注册，由 DispatcherConfig 自动装配。
 * 下一步（V5）可通过 ScriptEngineMessageHandler 支持动态脚本。
 */
public interface ThingModelMessageHandler extends MessageHandler {

    /**
     * 将设备原始数据转换为物模型 JSON 字符串。
     *
     * @param ctx 物模型上下文（deviceId / rawBytes / serializationType / channel）
     * @return 标准物模型 JSON 字符串；null 表示无需转换（透传原始数据）
     */
    String convertToThingModel(ThingModelContext ctx);

    @Override
    default byte supportMsgType() {
        return MsgType.BUSINESS;
    }

    @Override
    default void handle(ChannelHandlerContext ctx, String deviceId, MessagePacket packet) {
        ThingModelContext tmCtx = new ThingModelContext(deviceId, packet);
        String json = convertToThingModel(tmCtx);
        if (json != null) {
            // 将转换后的物模型 JSON 通过原有 Kafka 通道投递
            packet.setBody(json);
            packet.setRawBody(json.getBytes(StandardCharsets.UTF_8));
        }
        // 继续走原有 BusinessMessageHandler 的 Kafka 投递
    }
}
```

**无需修改核心代码**：`DispatcherConfig.java` 中 `@Bean MessageDispatcher(List<MessageHandler>)` 自动收集所有实现。

**预留 V5 接口**（本期不实施）：

## 四、风险审视

### 4.1 失败优先分析

| 特性 | 最可能失败原因 | 严重度 | 缓解措施 |
|------|---------------|--------|---------|
| HashedWheelTimer | Timer 线程未正确停止导致 JVM 无法退出 | 🔴 高 | @PreDestroy 调 timer.stop()，加 finally |
| Backpressure | 恢复 autoRead 的 channelWritabilityChanged 在高并发下丢失 | 🟡 中 | 同时使用定时心跳检查 writable 状态 |
| IP 过滤 | Redis 不可达时黑名单不可用 | 🟢 低 | 设计为 fail-open（放行），本地 Caffeine 兜底 |
| 优雅停机 | Kafka 遗言投递超时阻塞 JVM 退出 30s+ | 🟡 中 | 设置 `orTimeout(5s)`，超时则强制退出 |
| 集群路由 | Redis Pub/Sub 消息无持久化，指令丢失 | 🟡 中 | 增加客户端确认 + 超时重试机制 |
| Modbus | ProtocolId 非 0x0000 的设备误判为非 Modbus | 🟢 低 | 增加 ProtocolId 白名单配置 |
| OPC-UA | Milo 堆内存消耗过大导致 Gateway OOM | 🟡 中 | 独立 OpcUaServer 实例，限制 maxSessionCount |
| 物模型 | 插件实现质量参差不齐，转换异常导致 Kafka 垃圾数据 | 🟡 中 | 插件注册时执行冒烟测试 |

### 4.2 隐性成本

- **HashedWheelTimer**：tick 精度 100ms 导致实际超时 ±100ms 偏差，对 ACK 场景可接受
- **集群路由**：Redis Pub/Sub 无消息持久化，后续若要求指令不丢需改 gRPC（V5 成本）
- **OPC-UA**：Milo SDK ~5MB，OpcUaServer 额外 50MB+ 堆内存，小内存部署需评估
- **动态物模型（V5）**：脚本引擎带来安全沙箱问题，必须限制执行时间/内存/CPU

### 4.3 不可逆决策

| 决策 | 不可逆程度 | 说明 |
|------|-----------|------|
| Redis Hash 存储集群会话 | 🔴 高 | 上线后旧版监控/客户端依赖此结构，迁移成本大 |
| HashedWheelTimer tick 100ms | 🟢 低 | 配置项可调 |
| Pub/Sub 而非 gRPC 跨节点 | 🟡 中 | 后续改 gRPC 需新增 Proto 定义和独立端口 |

### 4.4 适用边界

**方案生效的前提**：
- Redis 可用（延迟 < 10ms）
- 集群节点间内网可达
- Kafka 可用（离线遗言投递）

**方案失效的场景**：
- Redis 宕机 → 集群路由不可用（降级为本地直发）
- Pub/Sub 消息大量积压 → 跨节点指令延迟不可控
- 单机 >10 万连接 → HashedWheelTimer tick 精度退化

**不应使用的场景**：
- 跨机房/跨区域部署（网络延迟不可控）
- 指令投递需要 100% 可靠性（应用 Kafka 持久化指令队列）
