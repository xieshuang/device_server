### 阶段 3：安全增强

**目标**：填补防重放 nonce 缓存和设备吊销两个安全缺口。

#### 3.1 Nonce 防重放缓存

**问题**：当前仅有 timestamp ±5min 校验，同一时间窗口内同一 token 可无限重放。

**方案**：AUTH_REQ 中增加 `nonce` 字段（客户端生成的随机字符串），服务端在 Redis 中缓存 `auth:nonce:{nonce}`，TTL = timestamp 容差窗口（5min），同一 nonce 不可重复使用。

**新增文件**：`com.xsh.netty.auth.NonceValidator.java`

```java
@Service
public class NonceValidator {
    private static final String NONCE_PREFIX = "auth:nonce:";

    private final StringRedisTemplate redisTemplate;

    /**
     * 校验 nonce 是否已使用。首次使用返回 true 并记录，重复使用返回 false。
     */
    public boolean validateAndRecord(String nonce, long ttlSeconds) {
        String key = NONCE_PREFIX + nonce;
        Boolean success = redisTemplate.opsForValue()
                .setIfAbsent(key, "1", ttlSeconds, TimeUnit.SECONDS);
        return Boolean.TRUE.equals(success);
    }
}
```

**修改文件**：`RedisAuthService.java` — authenticate 流程中先调用 nonceValidator.validateAndRecord()。

**AuthRequest 扩展**：`AuthRequest.java` 增加 `nonce` 字段（String，默认空串兼容旧客户端）。

**配置**：`netty.server.auth.nonce-enabled: true`

#### 3.2 设备吊销机制

**问题**：无运行时吊销设备的能力，需手动删 Redis key。

**方案**：Redis Key `device:revoked:{deviceId}`，鉴权时先检查是否在吊销列表，已吊销设备拒绝连接 + 记录日志。

**新增文件**：`com.xsh.netty.auth.DeviceRevocationService.java`

```java
@Service
public class DeviceRevocationService {
    private static final String REVOKED_PREFIX = "device:revoked:";

    private final StringRedisTemplate redisTemplate;

    /** 吊销设备（永久或指定时长） */
    public void revoke(String deviceId, long ttlSeconds) {
        redisTemplate.opsForValue().set(REVOKED_PREFIX + deviceId, 
                String.valueOf(System.currentTimeMillis()), ttlSeconds, TimeUnit.SECONDS);
    }

    /** 解除吊销 */
    public void unrevoke(String deviceId) {
        redisTemplate.delete(REVOKED_PREFIX + deviceId);
    }

    /** 检查设备是否已吊销 */
    public boolean isRevoked(String deviceId) {
        return Boolean.TRUE.equals(redisTemplate.hasKey(REVOKED_PREFIX + deviceId));
    }
}
```

**修改文件**：`RedisAuthService.java` — authenticate 流程中在 timestamp 校验后、Redis 查密钥前检查 isRevoked()。

### 阶段 4：消息追踪与配置加密

#### 4.1 消息全链路追踪

**目标**：deviceId + sequenceId + traceId 贯穿编解码→鉴权→Dispatcher→Kafka 全链路。

**修改文件**：
- `MessagePacket.java` — 新增 `String traceId` 字段
- `BusinessMessageHandler.java` — Kafka Header 增加 `traceId`
- `CustomProtocolDecoder.java` — 解码时生成 traceId（UUID 短格式）

**traceId 格式**：用 `UUID.randomUUID().toString().replace("-", "").substring(0, 12)` 生成 12 字符短 ID，兼顾可读性和唯一性。

#### 4.2 配置加密（Jasypt）

**目标**：application.yml 中 Redis 密码明文存储改为密文。

**Maven 依赖**：
```xml
<dependency>
    <groupId>com.github.ulisesbocchio</groupId>
    <artifactId>jasypt-spring-boot-starter</artifactId>
    <version>3.0.5</version>
</dependency>
```

**使用方式**：
```yaml
spring:
  data:
    redis:
      password: ENC(加密后的密文)
```

加密密钥通过环境变量 `JASYPT_ENCRYPTOR_PASSWORD` 或启动参数注入，不入库。

#### 4.3 Surefire 覆盖率配置

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-surefire-plugin</artifactId>
    <configuration>
        <!-- 运行所有测试，单个失败不中断 -->
        <failIfNoTests>false</failIfNoTests>
    </configuration>
</plugin>
```

## 七、风险审视

| 风险 | 严重度 | 缓解 |
|------|--------|------|
| nonce Redis 写入增加鉴权延迟 | 🟢 低 | SETNX 操作 < 1ms，可接受 |
| nonce 字段不向后兼容旧客户端 | 🟡 中 | 默认 null，无 nonce 的客户端降级为仅 timestamp 校验 |

| jasypt 密钥泄露 | 🟡 中 | 通过环境变量注入，不入库、不提交到 Git |
| 大量测试增加构建时间 | 🟢 低 | 预计从 3s → 10s，可接受 |
