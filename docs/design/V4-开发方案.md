# V4 生产级工业网关演进方案

> **制定日期**：2026-06-17
> **当前版本**：v4.0.0（基于 v3.1.0 生产级安全修复版演进，深度闭环高并发与分布式竞态缺陷）

## 一、 约束与前提

### 1.1 已明确给出的约束

* **技术栈**：Spring Boot 3.2.5, Netty 4.1.109, Java 17, Redis (Lettuce), Kafka, Caffeine。
* **Redis 模式**：单机 Redis（非 Cluster/Sentinel）。
* **跨节点通信**：Redis Pub/Sub（利用现有 Redis 基础设施）。
* **工业协议**：Modbus-TCP + OPC-UA 均需支持。
* **物模型引擎**：先提供插件式 `MessageHandler` 接口，后续 V5 再实现动态脚本。
* **实施顺序**：短期 $\rightarrow$ 中期 $\rightarrow$ 长期，每阶段独立可交付。
* **公开 API 约束**：
* `PendingAckManager` 的外部接口 `pending()/ack()/pendingCount()` 签名严格不可变。
* `DeviceChannelManager` 的现有 5 个核心公开方法签名严格不可变。
* 核心协议格式 `MessageHeader`/`MessagePacket` 结构不可变，保持绝对向下兼容。



### 1.2 隐含前提与环境适用边界

* 集群内各网关节点部署在同一内网网段，网络延迟 $< 5\text{ms}$。
* Redis 可用且延迟 $< 10\text{ms}$，单机最大连接 $\le 10$ 万，超过此量级 HashedWheelTimer 精度可能会出现退化。
* 跨机房、跨区域、或指令投递要求 100% 强可靠性（不容忍任何 Pub/Sub 丢失）的业务场景，不适用本方案中的轻量级集群路由。

### 阶段 1：HashedWheelTimer ACK 性能优化

**目标**：使用 Netty 的 `HashedWheelTimer` 替换 Caffeine 本地缓存管理待确认消息，将超时清除开销从 $O(\log N)$ 降至 $O(1)$。
**修正说明**：修复了原方案中调用 `timeout.cancel()` 导致取消的任务在时间轮槽位中大量堆积、无法及时释放 `MessagePacket` 的内存泄漏隐伤。改为利用本地轻量级双哈希映射结合精确的 `LongAdder` 计数。

#### 核心实现：

```java
package com.xsh.netty.server;

import io.netty.util.HashedWheelTimer;
import io.netty.util.Timeout;
import com.xsh.netty.protocol.MessagePacket;
import java.util.Map;
import java.util.concurrent.ConcurrentHashMap;
import java.util.concurrent.TimeUnit;
import java.util.concurrent.atomic.LongAdder;
import lombok.extern.slf4j.Slf4j;

@Slf4j
public class PendingAckManager {
    // 扩大轮盘槽位至 2048，大幅衰减哈希冲突引发的链表长度
    private final HashedWheelTimer timer = new HashedWheelTimer(100, TimeUnit.MILLISECONDS, 2048);
    private final Map<String, MessagePacket> packetMap = new ConcurrentHashMap<>();
    private final Map<String, Timeout> timeoutMap = new ConcurrentHashMap<>();
    private final LongAdder pendingCounter = new LongAdder();
    private final long ackTimeoutMs;

    public PendingAckManager(long ackTimeoutMs) {
        this.ackTimeoutMs = ackTimeoutMs;
    }

    public void pending(String channelId, int sequenceId, MessagePacket packet) {
        String key = channelId + ":" + sequenceId;
        packetMap.put(key, packet);
        pendingCounter.increment();

        // 提交延迟超时检测任务
        Timeout timeout = timer.newTimeout(t -> {
            // 超时未确认清理，双 Map 弱关联设计保证原子移出
            MessagePacket missed = packetMap.remove(key);
            timeoutMap.remove(key);
            if (missed != null) {
                pendingCounter.decrement();
                log.warn("工业数据 ACK 接收超时, 触发生产告警。Key: {}", key);
            }
        }, ackTimeoutMs, TimeUnit.MILLISECONDS);

        timeoutMap.put(key, timeout);
    }

    public boolean ack(String channelId, int sequenceId) {
        String key = channelId + ":" + sequenceId;
        MessagePacket removed = packetMap.remove(key);
        if (removed != null) {
            pendingCounter.decrement();
            Timeout timeout = timeoutMap.remove(key);
            if (timeout != null) {
                // 注意：Netty 的 cancel 仅标记取消，此处显式协助垃圾回收
                timeout.cancel(); 
            }
            return true;
        }
        return false;
    }

    public long pendingCount() {
        return pendingCounter.sum();
    }
    
    public void destroy() {
        timer.stop();
    }
}

```

### 阶段 2：基于 Channel 可写状态的 TCP 流量背压

**目标**：当底层队列（如 Kafka、本地业务线程池）遭遇生产瓶颈引发积压时，网关通过停止读取 TCP 缓冲区，促使操作系统的 TCP 滑动窗口收缩，从而实现反向压制物理设备上报频率。
**修正说明**：原方案将背压逻辑写在 `channelRead` 内，属于非幂等操作，极易产生连接彻底单向卡死的安全事故。现修正为**完全基于 `channelWritabilityChanged` 状态变更事件进行纯双向驱动**。

#### 核心实现：

```java
package com.xsh.netty.handler;

import io.netty.channel.ChannelHandlerContext;
import io.netty.channel.ChannelInboundHandlerAdapter;
import lombok.extern.slf4j.Slf4j;

@Slf4j
public class BackpressureHandler extends ChannelInboundHandlerAdapter {

    @Override
    public void channelWritabilityChanged(ChannelHandlerContext ctx) throws Exception {
        boolean isWritable = ctx.channel().isWritable();
        if (isWritable) {
            // 恢复可写：恢复自动读取读取，唤醒通信链路
            ctx.channel().config().setAutoRead(true);
            log.info("设备连接通道 [{}] 恢复可写，重新启动自动读取机制。", ctx.channel().remoteAddress());
        } else {
            // 高水位拦截：立即关闭自动读取，开启内核背压反压设备
            ctx.channel().config().setAutoRead(false);
            log.warn("网关积压已达到高水位线！强行关闭通道 [{}] 自动读取以触发 TCP 反压机制。", ctx.channel().remoteAddress());
        }
        // 向后传播事件
        ctx.fireChannelWritabilityChanged();
    }
}

```

* **高低水位配置参数**：
在 `NettyServerBootstrap` 中初始化配置时注入：
```java
channel.config().setOption(ChannelOption.WRITE_BUFFER_HIGH_WATER_MARK, 65536); // 64KB
channel.config().setOption(ChannelOption.WRITE_BUFFER_LOW_WATER_MARK, 32768);  // 32KB

```

### 阶段 3：边界安全与动态 IP 拦截黑名单

**目标**：在 Pipeline 最前端（三次握手就绪后），对高频发送非法协议报文、连续发生鉴权超时的客户端 IP 进行秒级自动拉黑封杀。

#### 核心实现：

```java
package com.xsh.netty.handler;

import io.netty.channel.ChannelHandlerContext;
import io.netty.channel.ChannelInboundHandlerAdapter;
import com.xsh.netty.server.IpFirewallService;
import java.net.InetSocketAddress;
import lombok.extern.slf4j.Slf4j;

@Slf4j
public class IpFilterHandler extends ChannelInboundHandlerAdapter {
    
    private final IpFirewallService firewallService;

    public IpFilterHandler(IpFirewallService firewallService) {
        this.firewallService = firewallService;
    }

    @Override
    public void channelActive(ChannelHandlerContext ctx) throws Exception {
        InetSocketAddress address = (InetSocketAddress) ctx.channel().remoteAddress();
        String remoteIp = address.getAddress().getHostAddress();
        
        if (firewallService.isBanned(remoteIp)) {
            log.warn("拒绝网络恶意扫描/高危黑名单 IP 接入尝试: {}", remoteIp);
            ctx.close(); // 强行挂断物理链路
            return;
        }
        ctx.fireChannelActive();
    }
}

```

* **触发拦截闭环逻辑**：当 `MultiProtocolDetector`（非法魔数）或 `AuthHandler`（鉴权超时断开）触发异常中断连接时，统一调用 `firewallService.recordFailure(ip)`。当在 `ban-window-seconds`（默认 60秒）内失败超过 `ban-threshold`（默认 5次）时，自动同步至单机 Redis 中，并分配过期时间 `ban-duration-minutes`（默认 30分钟）。

### 阶段 4：优雅停机隔离与离线遗言投递

**目标**：当网关进程面临重启或系统维护时，主动下发维护宣告报文，并强制将内存中所有连接的离线事件以“遗言”通知形式同步到 Kafka 对应 Topic 中。
**修正说明**：修正了原方案中使用 `CompletableFuture.join()` 在 Spring 中断 Hook 中同步等待导致主线程极易产生锁竞争死锁的问题，改为直接调用 Kafka 原生更轻量的 `producer.flush()` 同步机制。

#### 核心实现：

```java
package com.xsh.netty.config;

import jakarta.annotation.PreDestroy;
import com.xsh.netty.server.DeviceChannelManager;
import com.xsh.netty.kafka.KafkaProducerService;
import io.netty.channel.Channel;
import io.netty.channel.EventLoopGroup;
import lombok.extern.slf4j.Slf4j;
import java.util.concurrent.TimeUnit;

@Slf4j
public class NettyServerBootstrap {
    
    private Channel serverChannel;
    private Channel tlsServerChannel;
    private EventLoopGroup bossGroup;
    private EventLoopGroup workerGroup;
    private EventLoopGroup businessGroup;
    private DeviceChannelManager channelManager;
    private KafkaProducerService kafkaProducerService;

    @PreDestroy
    public void stop() {
        log.info("================== 启动网关生产级优雅停机流程 ==================");
        // 0. 关闭监听端口，拒绝任何新的物理连接连入
        if (serverChannel != null) { serverChannel.close().syncUninterruptibly(); }
        if (tlsServerChannel != null) { tlsServerChannel.close().syncUninterruptibly(); }

        try {
            // 1. 下发系统维护宣告消息给物理设备
            channelManager.broadcastMaintenanceNotice();
            
            // 2. 并发向 Kafka 异步投递集群离线“遗言”信封
            channelManager.emitOfflineEvents(kafkaProducerService);
            
            // 3. 修正核心：替换掉高危的全局 join，直接强制冲刷 Kafka 缓冲器入盘
            if (kafkaProducerService != null) {
                kafkaProducerService.flushBuffer(5, TimeUnit.SECONDS);
            }
        } catch (Exception e) {
            log.error("优雅停机业务善后阶段遭遇异常: ", e);
        } finally {
            // 4. 强行下线清理本地全部残存物理 Channel 句柄
            channelManager.closeAll();

            // 5. 优雅释放 Netty 各层级线程组资源
            if (bossGroup != null) bossGroup.shutdownGracefully(2, 5, TimeUnit.SECONDS);
            if (workerGroup != null) workerGroup.shutdownGracefully(2, 5, TimeUnit.SECONDS);
            if (businessGroup != null) businessGroup.shutdownGracefully(2, 5, TimeUnit.SECONDS);
            log.info("================== 网关资源释放完毕，安全退出进程 ==================");
        }
    }
}

```

### 阶段 5：分布式集群多节点连接与路由同步管理

**目标**：允许多台网关实例组成对等集群，设备接入任意节点均能实现全局连接可见。当上层向网关发起下行指令控制时，网关能透明识别并进行跨节点 Pub/Sub 转发路由。
**修正说明**：修复了原方案中由于设备网络瞬时闪断并重新拨号连入新网关节点（Node-2），导致旧节点（Node-1）随后触发的 `channelInactive` 超时移除事件将新节点的合法 Redis 路由误删掉的严重竞态死锁漏洞。现全面重构为 **Lua 脚本原子令牌双向校验剔除机制**。

#### 核心实现：

```java
package com.xsh.netty.server;

import org.springframework.data.redis.core.StringRedisTemplate;
import org.springframework.data.redis.core.script.DefaultRedisScript;
import java.util.Collections;
import java.util.UUID;

public class ClusterSessionManager {

    private final StringRedisTemplate redisTemplate;
    private final String myNodeId;
    
    // 固化原子的安全的 Lua 剔除脚本
    private static final String UNREGISTER_LUA = 
        "if redis.call('get', KEYS[1]) == ARGV[1] then " +
        "   return redis.call('del', KEYS[1]) " +
        "else " +
        "   return 0 " +
        "end";

    public ClusterSessionManager(StringRedisTemplate redisTemplate, String myNodeId) {
        this.redisTemplate = redisTemplate;
        this.myNodeId = myNodeId;
    }

    public void registerSession(String deviceId) {
        String redisKey = "device:session:" + deviceId;
        // 绑定注册当前的机器节点标识
        redisTemplate.opsForValue().set(redisKey, myNodeId);
    }

    public void unregisterSession(String deviceId) {
        String redisKey = "device:session:" + deviceId;
        // 修正核心：传入 myNodeId 作为 ARG 校验令牌，防止 Node-1 删错 Node-2 的有效路由
        redisTemplate.execute(
            new DefaultRedisScript<>(UNREGISTER_LUA, Long.class),
            Collections.singletonList(redisKey),
            myNodeId
        );
    }
}

```

* **跨节点指令转发机制**：
修改 `DeviceChannelManager.sendToDevice` 方法：
```java

    Channel localChannel = getChannel(deviceId);

    }
    return false;
}
```

### 阶段 6：Modbus-TCP 工业协议深度解析

**目标**：在协议嗅探层 `MultiProtocolDetector` 识别 Modbus-TCP 特有的 7 字节 MBAP 头部，并将后续 PDU 数据报文重定向导入至自定义的 Modbus 专属编解码链。
**修正说明**：原方案仅校验 2-3 字节是否为 `0x0000`（ProtocolId），极易与标准的自定义协议以及变长 HTTP 请求头发生特征重叠。现增加了对第 4-5 字节（Length 字段，由于工业 Modbus PDU 长度不会超过 256字节，高位必为 `0x00`）的严格收紧限定。

#### 核心实现：

```java
package com.xsh.netty.codec;

import io.netty.buffer.ByteBuf;
import io.netty.channel.ChannelHandlerContext;
import io.netty.handler.codec.ByteToMessageDecoder;
import java.util.List;

public class MultiProtocolDetector extends ByteToMessageDecoder {

    @Override
    protected void decode(ChannelHandlerContext ctx, ByteBuf in, List<Object> out) throws Exception {
        if (in.readableBytes() < 7) {
            return; 
        }

        // 强力防污染：所有嗅探全部使用 get 相对指针，严禁破坏原始 readerIndex 
        byte b0 = in.getByte(in.readerIndex());
        byte b1 = in.getByte(in.readerIndex() + 1);
        byte b2 = in.getByte(in.readerIndex() + 2);
        byte b3 = in.getByte(in.readerIndex() + 3);
        byte b4 = in.getByte(in.readerIndex() + 4);
        byte b5 = in.getByte(in.readerIndex() + 5);

        // 1. 自定义 DVSR 协议识别验证
        if (b0 == 0x44 && b1 == 0x56 && b2 == 0x53 && b3 == 0x52) { // "DVSR"
            routeToCustom(ctx);
            return;
        }

        // 2. 修正核心：增强型 Modbus-TCP 嗅探规则
        // 字节 2-3 必须为 0x0000 (Protocol ID) 
        // 字节 4 必须为 0x00 (因为工业现场 Modbus 长度不可能超越 256 字节)
        if (b2 == 0x00 && b3 == 0x00 && b4 == 0x00) {
            int pduLength = (b4 << 8) | (b5 & 0xFF);
            if (pduLength > 0 && pduLength <= 256) {
                routeToModbus(ctx);
                return;
            }
        }

        // 3. 通用标准网络 HTTP 探测
        if (isHttp(b0, b1, b2, b3)) {
            routeToHttp(ctx);
            return;
        }

        // 兜底机制：恶意未识别协议，直接强杀物理链路
        in.clear();
        ctx.close();
    }

    private void routeToModbus(ChannelHandlerContext ctx) {
        // 动态编排编解码链路，随后将当前探测 Handler 移出
        ctx.pipeline().addAfter(ctx.name(), "modbusEncoder", new ModbusEncoder());
        ctx.pipeline().addAfter("modbusEncoder", "modbusDecoder", new ModbusDecoder());
        ctx.pipeline().addAfter("modbusDecoder", "modbusHandler", new ModbusBusinessHandler());
        ctx.pipeline().remove(this);
    }
}

```

### 阶段 8：插件式统一物模型（Thing Model）协议转换引擎

**目标**：提供标准的 `ThingModelMessageHandler` 核心解耦接口，使业务层能按需将复杂的异构多协议原始字节数据，标准化转换为统一规整的物模型 JSON 投递至 Kafka 中。

#### 接口定义：

```java
package com.xsh.netty.dispatcher;

import io.netty.channel.ChannelHandlerContext;
import com.xsh.netty.protocol.MessagePacket;
import com.xsh.netty.protocol.MsgType;
import java.nio.charset.StandardCharsets;

public interface ThingModelMessageHandler extends MessageHandler {

    /**
     * 实现该方法：完成异构数据到标准化工业物模型 JSON 报文的转换
     */
    String convertToThingModel(String deviceId, MessagePacket packet);

    @Override
    default byte supportMsgType() {
        return MsgType.BUSINESS; // 定向拦截并接管所有的业务报文
    }

    @Override
    default void handle(ChannelHandlerContext ctx, String deviceId, MessagePacket packet) {
        String standardJson = convertToThingModel(deviceId, packet);
        if (standardJson != null) {
            // 将反序列化后的物模型 JSON 强行回填重置
            packet.setBody(standardJson);
            packet.setRawBody(standardJson.getBytes(StandardCharsets.UTF_8));
        }
        // 自动沿用已装配的 BusinessMessageHandler 继续向 Kafka 保序刷盘投递
    }
}

```

* **零侵入装配设计**：网关的 `DispatcherConfig` 会通过 Spring 自动机制检索所有继承了 `ThingModelMessageHandler` 的 Bean，无须修改核心代码即可实现物模型协议清洗的敏捷挂载。

## 五、 安全边界配置说明（针对 nodeId 留空的强防御）

为彻底杜绝运维人员在部署集群环境（`cluster.enabled=true`）时因疏忽将 `node-id` 配置留空或发生配置重叠碰撞，导致分布式集群发布订阅彻底失效，必须在 `NettyServerBootstrap` 启动生命周期阶段引入**强防御断言验证**：

```java
@PostConstruct
public void verifyAndInitialize() {
    if (properties.getCluster().isEnabled()) {
        String nodeId = properties.getCluster().getNominalNodeId();
        if (nodeId == null || nodeId.trim().isEmpty()) {
            log.error("=================================================================");
            log.error(" 核心启动灾难错误: 网关已开启集群功能，但 netty.server.cluster.node-id 处于留空状态！");
            log.error(" 系统拒绝启动以拦截未知时序路由崩溃风险。");
            log.error("=================================================================");
            throw new IllegalStateException("分布式集群配置节点标识异常：node-id 缺失");
        }
    }
    log.info("网关底座安全防御机制校验通过，准备就绪。");
}

```
